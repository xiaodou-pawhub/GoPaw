## 问题描述

重构前，`ListLogs` 接口采用 `bufio.Scanner` 顺序扫描全量日志文件后再截取末端。在生产环境下，当日志文件体积增长（如达到 GB 级）时，该接口会导致严重的磁盘 IO 负载和 CPU 占用，并伴随明显的响应延迟。

---

## 修复方案

### 1. 实现反向块读取算法 (Tail)
放弃顺序扫描，改为基于文件指针（Seek）的反向读取逻辑：
- **按块步进**: 每次从文件末尾向前移动 4KB 步长。
- **内存友好**: 仅在内存中处理当前读取的字节块，不加载文件全量内容。
- **即时终止**: 达到用户请求的 `limit` 行数后立即停止 IO，确保大文件场景下的响应速度始终为毫秒级。

### 2. 健壮性增强
- **残行处理**: 实现了 `leftover` 机制，正确处理跨块截断的日志行。
- **边界保护**: 完善了文件为空、文件大小小于缓冲区、文件读取错误等边界条件的降级处理。

### 3. 安全脱敏
- 将脱敏逻辑抽离为 `isSensitive` 纯函数，确保日志输出符合安全审计要求。

```go
// 核心优化逻辑片段
for cursor > 0 && len(result) < limit {
    cursor -= readSize
    file.Seek(cursor, io.SeekStart)
    file.Read(buffer)
    // 从后向前解析块内容...
}
```
